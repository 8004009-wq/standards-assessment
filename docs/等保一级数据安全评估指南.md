# 等保一级数据安全评估指南

## 📋 标准依据

- **标准名称**: 信息安全技术 网络安全等级保护数据安全基本要求
- **标准编号**: GA/T 2380-2026
- **标准类型**: 公共安全行业标准
- **保护级别**: 第一级（自主保护级）
- **适用范围**: 一般信息系统

---

## 🎯 一级系统判定标准

符合以下条件的系统可定为一级：

- [ ] 不涉及国家安全、社会秩序和公共利益
- [ ] 遭到破坏后对公民、法人和其他组织合法权益造成一般损害
- [ ] 用户数量较少（通常低于 10 万）
- [ ] 数据处理规模较小
- [ ] 系统服务范围有限

## ✅ 评估要求详解

### 4.1 数据分类分级

**要求**: 应建立数据分类分级管理制度，明确数据分类分级方法和管理要求

**评估要点**:
- 是否制定数据分类分级管理制度
- 是否明确数据分类方法（如按业务、按敏感度等）
- 是否明确数据分级标准（如一般、重要、核心）
- 是否明确不同级别数据的管理要求

**证据示例**:
- 《数据分类分级管理制度》文档
- 数据分类分级清单
- 数据资产目录

---

### 4.2 数据采集安全

**要求**: 应明确数据采集的目的、方式和范围，遵循合法、正当、必要的原则

**评估要点**:
- 是否明确数据采集目的
- 是否明确数据采集方式
- 是否明确数据采集范围
- 是否遵循合法、正当、必要原则

**证据示例**:
- 数据采集管理制度
- 隐私政策
- 用户授权同意记录
- 数据采集清单

---

### 4.3 数据传输安全

#### 4.3 a) 重要数据传输前加密

**要求**: 应在重要数据传输前进行加密处理

**评估要点**:
- 是否识别重要数据
- 是否在传输前对重要数据加密
- 加密算法是否安全可靠

**证据示例**:
- 加密配置截图
- 加密算法说明
- 传输数据包分析

#### 4.3 b) 采用安全传输协议

**要求**: 应采用安全的传输协议进行数据传输

**评估要点**:
- 是否使用 HTTPS/TLS 等安全协议
- 是否禁用不安全的传输协议（如 HTTP、FTP）
- TLS 版本是否符合要求（建议 TLS 1.2+）

**证据示例**:
- 网络配置截图
- SSL/TLS 配置
- 协议分析报告

---

### 4.4 数据存储安全

#### 4.4 a) 重要数据存储加密

**要求**: 应对重要数据存储进行加密保护

**评估要点**:
- 是否识别重要数据
- 是否对重要数据加密存储
- 加密密钥是否安全管理

**证据示例**:
- 数据库加密配置
- 文件加密截图
- 密钥管理制度

#### 4.4 b) 建立数据备份机制

**要求**: 应建立数据备份机制，定期备份重要数据

**评估要点**:
- 是否建立数据备份制度
- 是否定期执行备份
- 备份数据是否可恢复

**证据示例**:
- 《数据备份管理制度》
- 备份策略配置
- 备份执行记录
- 恢复测试记录

---

### 4.5 数据使用安全

**要求**: 应建立数据访问控制机制，根据业务需要分配数据访问权限

**评估要点**:
- 是否建立访问控制制度
- 是否根据业务需要分配权限
- 是否遵循权限最小化原则
- 是否有权限审批流程

**证据示例**:
- 《数据访问控制管理制度》
- 权限分配记录
- 权限审批流程
- 系统权限配置截图

---

### 4.6 数据交换共享安全

**要求**: 应在数据交换共享前与接收方签订安全协议，明确安全责任和保护要求

**评估要点**:
- 是否与接收方签订安全协议
- 协议是否明确安全责任
- 协议是否明确数据保护要求

**证据示例**:
- 数据安全协议模板
- 已签署的安全协议
- 数据共享清单
- 接收方资质审查记录

---

### 4.7 数据销毁安全

**要求**: 应建立数据销毁管理制度，采用技术手段确保数据无法被恢复

**评估要点**:
- 是否建立数据销毁制度
- 是否采用技术手段销毁数据
- 销毁后数据是否无法恢复

**证据示例**:
- 《数据销毁管理制度》
- 数据销毁记录
- 销毁工具说明
- 销毁过程记录

---

### 4.8 安全事件处置

**要求**: 应建立数据安全事件应急处置机制，及时响应和处置数据安全事件

**评估要点**:
- 是否建立应急处置机制
- 是否制定应急预案
- 是否及时响应和处置事件

**证据示例**:
- 《数据安全事件应急预案》
- 应急响应流程
- 安全事件记录
- 应急演练记录

## 📈 合规判定

### 合规率计算

```
合规率 = (符合项数 + 部分符合项数 × 0.5) ÷ 适用项总数 × 100%
```

### 一级系统合规要求

| 指标 | 要求 |
|------|------|
| 总体合规率 | ≥80% |
| 评估项总数 | 10 项 |
| 最低符合项数 | 8 项（如全部符合） |

### 合规判定示例

```
总评估项：10 项
- 符合：8 项（8 × 5 = 40 分）
- 部分符合：1 项（1 × 2.5 = 2.5 分）
- 不符合：1 项（0 分）
- 不适用：0 项

总分：42.5 分
满分：50 分
合规率：42.5 ÷ 50 × 100% = 85%

判定：☑ 通过（≥80%）
```

## 📝 评估流程

### 第一阶段：准备（1 周）

```
□ 确定评估范围和边界
□ 收集制度文档和技术资料
□ 准备评估工具和检查表
```

### 第二阶段：评估（1-2 周）

```
□ 逐项评估 10 个评估项
□ 收集证据材料
□ 记录不符合项
□ 计算合规率
```

### 第三阶段：整改（2-4 周）

```
□ 制定整改计划
□ 实施技术和管理整改
□ 验证整改效果
```

### 第四阶段：复测（1 周）

```
□ 对不符合项进行复测
□ 计算最终合规率
□ 生成评估报告
```

## 🔄 持续改进

### 一级系统监督要求

| 项目 | 要求 |
|------|------|
| **安全自查** | 每年至少 1 次 |
| **风险评估** | 每 2 年至少 1 次 |
| **安全培训** | 每年至少 1 次 |

### 变更管理

发生以下变更时需重新评估：

- [ ] 系统架构重大调整
- [ ] 数据处理规模大幅增长
- [ ] 业务范围发生重大变化
- [ ] 发生数据安全事件
